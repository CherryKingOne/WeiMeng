# 前端“无等待感”交互体验优化指南

## 目标
把“接口请求实现”升级为“产品级交互体验方案”，核心是从阻断式等待转为非阻塞同步：
- 用户操作后先获得即时反馈
- 数据在后台同步
- 只做差异化更新，不打断上下文

## 触发后默认工作流
1. 识别当前页面属于哪类场景：首屏、搜索/筛选、分页、新增、删除、状态切换、页面切回。
2. 先做体验层诊断，再给技术方案；避免只给“加 loading”。
3. 输出“策略 + 状态机 + 落地步骤 + 验收标准”四段式方案。
4. 如用户要求改代码，直接按该方案在代码中落地，并说明对应策略映射关系。

## 四层优化框架

### 第一层：视觉连贯性（消除闪烁与白屏）
- 首屏无缓存时显示骨架屏。
- 有旧数据时刷新不清空列表，保留旧数据占位。
- 将“首屏加载”和“后台刷新”分离为不同状态，不复用同一个阻断 loading。

### 第二层：后台静默同步（消除刷新感）
- 增删改后优先标记数据过期，不立即清空。
- 后台静默拉取新列表，利用框架 diff 做最小变更。
- 保持滚动位置、筛选条件、展开状态不丢失。

### 第三层：乐观更新（消除延迟感）
- 高频且高成功率操作默认乐观更新：先改 UI，再发请求。
- 操作前保存快照，失败时回滚并提示。
- 不对高风险事务（资金、权限、强一致业务）盲目使用乐观策略。

### 第四层：缓存与生命周期（避免冗余请求）
- 按业务新鲜度决定是否启用聚焦自动刷新。
- 数据按复用价值进行缓存回收：短生命周期数据及时清理，可复用数据保留。
- 页面来回切换优先命中缓存，避免重复全量请求。

## 前端实现约束（React/Next 默认）
- 列表刷新时禁止 `setList([])` 再请求。
- 使用显式请求模式，例如：`initial | refresh | silent`。
- `initial` 才能触发骨架；`refresh` 只显示微弱同步标识；`silent` 不打断 UI。
- 新增场景优先“先插入再静默校准”；删除/切换优先“先变更再失败回滚”。
- 错误展示分层：
  - 无数据且失败：阻断错误卡 + 重试。
  - 有旧数据且失败：非阻断告警，不清空内容。

## 业务场景策略映射
- 首次进入：骨架屏。
- 搜索/筛选/分页：旧数据占位 + 局部弱加载。
- 新增/上传：成功反馈 + 静默刷新（或乐观插入 + 静默校准）。
- 删除/状态切换：乐观更新 + 失败回滚。
- 标签页切换：缓存优先，按需刷新。

## 输出模板
触发本 Skill 时，优先按以下格式输出：

### 1. 体验问题诊断
- 当前阻断点
- 用户可见抖动点
- 状态丢失风险

### 2. 优化策略组合
- 使用哪些层（1~4层）
- 每一层解决什么问题
- 取舍与边界

### 3. 前端状态机与请求时序
- 状态定义（例如 `initial/refresh/silent`）
- 操作时序（用户动作 -> UI反馈 -> 后台同步 -> 差异更新）
- 失败路径与回滚路径

### 4. 验收清单
- 是否消除白屏/闪烁
- 是否保留上下文（滚动、筛选、输入）
- 操作反馈是否即时
- 异常时是否可恢复

## 团队对齐口径
在需要面向 PM/UI 说明时，统一表达为：
- 前端体验目标不是“更快请求”，而是“更少等待感”。
- 用户不应感知完整请求周期，只应感知状态渐进更新。
