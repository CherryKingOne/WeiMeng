

### 1. 数据库设计 (PostgreSQL)

由于`shot_uuid`要求是20位纯数字，且不再进行数学运算，建议使用 `VARCHAR(20)` 作为主键。内容字段使用 Postgres 的 `TEXT` 类型，它支持无限长度的字符串，非常适合存储 LLM 的回复。

```sql
-- 创建分镜/剧本生成记录表
CREATE TABLE shots (
    -- 20位纯数字ID，主键
    shot_uuid VARCHAR(20) PRIMARY KEY,
    
    -- 剧本库ID
    library_id VARCHAR(255) NOT NULL,
    
    -- 剧本文件ID
    script_id VARCHAR(255) NOT NULL,
    
    -- 核心存储字段：这里存储LLM生成的纯文本（包含JSON）
    -- 对应接口中的 'text'
    content TEXT NOT NULL, 
    
    -- 创建时间
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    
    -- 更新时间（如果用户修改了内容）
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

-- 为了加快查询速度，给 script_id 建索引
CREATE INDEX idx_shots_script_id ON shots(script_id);
```

### 2. 接口设计 (RESTful API)

**基础路由**: `/v4/shot`

#### 2.1 新增/保存 (Create) - POST
用户在前端对LLM生成的表格满意点击“保存”时调用。

*   **URL**: `/v4/shot`
*   **Method**: `POST`
*   **Request Body (JSON)**:
    ```json
    {
      "library_id": "lib_123",
      "script_id": "file_456",
      "text": "这里是LLM生成的长文本，里面可能包含 ```json [...] ``` 这样的内容"
    }
    ```
*   **后端逻辑**:
    1.  生成一个 **20位纯数字** 的 `shot_uuid` (算法见下文)。
    2.  执行 SQL `INSERT INTO shots ...`。
    3.  返回构建好的对象。
*   **Response**:
    ```json
    {
      "code": 200,
      "data": {
        "shot_uuid": "12345678901234567890",
        "library_id": "lib_123",
        "script_id": "file_456",
        "text": "这里是LLM生成的长文本...",
        "created_at": "2023-10-27T10:00:00Z"
      }
    }
    ```

#### 2.2 修改/更新 (Update) - PUT
用户后续如果修改了表格内容（比如在前端改了JSON里的某个字段），前端把整个对象转回字符串，重新发给后端更新。

*   **URL**: `/v4/shot` (或者 `/v4/shot?shot_uuid=...`，通常建议把ID放在Body或Path里，这里假设传在Body里或Query里)
    *   建议设计为: `/v4/shot` (通过Body识别) 或 `/v4/shot/:shot_uuid`
*   **Method**: `PUT`
*   **Request Body**:
    ```json
    {
      "shot_uuid": "12345678901234567890", 
      "text": "这是修改后的文本内容..."
    }
    ```
*   **SQL**: `UPDATE shots SET content = ?, updated_at = NOW() WHERE shot_uuid = ?`
*   **Response**: 返回更新后的完整数据。

#### 2.3 查询详情 (Read) - GET
用户再次进入该页面，或者点击历史记录回显时调用。

*   **URL**: `/v4/shot?shot_uuid=12345678901234567890`
*   **Method**: `GET`
*   **SQL**: `SELECT * FROM shots WHERE shot_uuid = ?`
*   **Response**:
    ```json
    {
        "shot_uuid": "12345678901234567890",
        "library_id": "lib_123",
        "script_id": "file_456",
        "text": "这里是数据库里存的原始文本...",
        "created_at": "..."
    }
    ```

#### 2.4 删除 (Delete) - DELETE
*   **URL**: `/v4/shot?shot_uuid=12345678901234567890`
*   **Method**: `DELETE`
*   **SQL**: `DELETE FROM shots WHERE shot_uuid = ?`

### 3. 核心代码逻辑实现思路 (伪代码/Node.js风格)

这里主要解决你疑惑的“怎么存”和“20位ID怎么生成”的问题。

#### 工具函数：生成20位纯数字ID
```javascript
function generate20DigitId() {
    // 简单实现：使用当前时间戳(13位) + 7位随机数
    // 注意：高并发下建议使用 Snowflake 算法，但在一般场景下这个够用了
    const timestamp = Date.now().toString(); // 13位
    let randomPart = '';
    for (let i = 0; i < 7; i++) {
        randomPart += Math.floor(Math.random() * 10).toString();
    }
    return timestamp + randomPart; 
}
```

#### 后端处理流程 (POST /v4/shot)
```javascript
// 假设框架为 Express/Koa 或类似

async function createShot(req, res) {
    // 1. 获取前端传入参数
    const { library_id, script_id, text } = req.body;

    // 2. 校验参数
    if (!library_id || !script_id || !text) {
        return res.status(400).json({ error: "参数缺失" });
    }

    // 3. 生成 20位 ID
    const shotUuid = generate20DigitId();
    const createdAt = new Date();

    try {
        // 4. 存入 PostgreSQL
        // 注意：数据库字段是 content，但接口交互是 text，这里要做映射
        const sql = `
            INSERT INTO shots (shot_uuid, library_id, script_id, content, created_at)
            VALUES ($1, $2, $3, $4, $5)
        `;
        await db.query(sql, [shotUuid, library_id, script_id, text, createdAt]);

        // 5. 返回结果 (按你的要求返回指定字段)
        return res.json({
            shot_uuid: shotUuid,
            library_id: library_id,
            script_id: script_id,
            text: text, // 返回存储的纯文本
            created_at: createdAt
        });

    } catch (error) {
        console.error(error);
        return res.status(500).json({ error: "存储失败" });
    }
}
```

### 4. 前端回显的逻辑（如何解析回来）

既然你在存的时候存的是 `txt`（可能包含 `{...}` json 字符串，也可能包含 LLM 的废话），取出来的时候逻辑应该和刚生成时保持一致：

1.  **前端调用 GET 接口**，拿到 `text` 字段的字符串。
2.  **前端复用你现有的“提取JSON”逻辑**。
    *   比如你之前是用正则表达式 `text.match(/```json([\s\S]*?)```/)` 或者寻找 `{` 和 `}` 的位置来提取。
    *   **关键点**：因为你存的是纯文本，所以无论JSON结构怎么变，只要符合JSON语法，前端 `JSON.parse()` 都能解析出来渲染成表格。

### 5. 总结你的疑问

*   **Q: 接口怎么设计？**
    *   A: 采用标准的 RESTful 风格，POST存，GET取，PUT改，DELETE删。输入输出严格遵守你定义的 `text` 字段名。
*   **Q: SQL语句怎么存储？**
    *   A: 使用 `INSERT INTO`，将文本内容放入 PostgreSQL 的 `TEXT` 类型字段中。不要试图用数据库去理解JSON，把它当成普通字符串处理。
*   **Q: 20位纯数字ID怎么办？**
    *   A: 后端在插入前通过算法（时间戳+随机数）生成字符串类型的数字，存入 `VARCHAR(20)` 主键。

这种设计方案最大程度地保留了灵活性（支持自定义JSON），同时满足了你对持久化存储和特定ID格式的要求。
