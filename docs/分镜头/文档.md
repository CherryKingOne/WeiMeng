这是一个非常典型的将非结构化/半结构化（JSON）数据转化为结构化关系型数据库（SQL）的需求。

为了满足你的要求（**完全拆分JSON、规范表名、支持角色复用、可视化友好、生成CRUD接口**），我设计了一套基于 **PostgreSQL** 的数据库架构。

### 1. 数据库设计分析与优化思路

**核心原则：**
1.  **拒绝 JSON 字段**：所有嵌套对象（如 `visualScene`、`characterInfo`、`prompts`）全部打散为关联表或字段。
2.  **表名规范**：统一使用 `script_` 作为前缀，方便在数据库工具中聚合查看。
3.  **角色复用逻辑**：
    *   创建一个 **`script_characters`（角色库表）**：仅存储角色的核心身份（姓名、性别、所属剧本库ID）。这样同一个剧本库下的不同章节（File）可以引用同一个角色ID。
    *   创建一个 **`script_shot_characters`（镜头角色详情表）**：存储该角色在**特定镜头**中的具体表现（外观描述、提示词、生成的图片）。
    *   **逻辑**：当解析JSON时，先去`script_characters`查找该剧本库下有没有叫这个名字的人，有则取ID，无则创建。然后将本次镜头中的衣服、提示词存入`script_shot_characters`。这样既保证了角色身份统一，又允许角色在第10场换衣服（数据独立）。

### 2. SQL 表结构设计 (PostgreSQL)

我们将创建 6 张表来完整承载这些数据。

```sql
-- 1. 剧本文件表 (对应 JSON 顶层 fileInfo 和 globalConfig)
-- 存储章节/文件维度的信息
CREATE TABLE script_files (
    id BIGSERIAL PRIMARY KEY,
    library_id BIGINT NOT NULL,          -- 关联外部剧本库ID (例如 6250527834723381)
    file_identifier VARCHAR(100),        -- 对应 fileId
    file_name VARCHAR(255),              -- 对应 fileName
    total_words INT,                     -- 对应 totalWords
    visual_style VARCHAR(100),           -- 对应 globalConfig.visualStyle
    generation_time TIMESTAMP WITH TIME ZONE, -- 对应 scriptGenerationTime
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
COMMENT ON TABLE script_files IS '剧本章节文件表';

-- 2. 镜头基础表 (对应 scriptShotList 的基础字段)
-- 存储镜号、文本、时长等核心信息
CREATE TABLE script_shots (
    id BIGSERIAL PRIMARY KEY,
    file_id BIGINT NOT NULL REFERENCES script_files(id) ON DELETE CASCADE, -- 关联文件
    shot_number INT NOT NULL,            -- 镜号
    original_text TEXT,                  -- 原始内容
    original_word_count INT,             -- 字数
    shot_type VARCHAR(50),               -- 类型 (对话/旁白等)
    duration VARCHAR(20),                -- 时长 (3.5s)
    context_summary TEXT,                -- 上下文总结
    scene_description_text TEXT,         -- 对生成的画面进行描述
    core_concept TEXT,                   -- 核心概念提取
    memo TEXT,                           -- 备注 (新增字段，用于筛选或标记)
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_shots_file_id ON script_shots(file_id);
COMMENT ON TABLE script_shots IS '镜头基础信息表';

-- 3. 角色注册表 (用于复用角色ID)
-- 同一个 library_id 下，角色名唯一
CREATE TABLE script_characters (
    id BIGSERIAL PRIMARY KEY,
    library_id BIGINT NOT NULL,          -- 归属的剧本库
    name VARCHAR(100) NOT NULL,          -- 角色名 (如: 林风眠)
    gender VARCHAR(20),                  -- 性别
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(library_id, name)             -- 保证同一剧本库下角色名不重复
);
COMMENT ON TABLE script_characters IS '角色身份注册表(用于复用)';

-- 4. 镜头角色详情表 (对应 characterInfo)
-- 记录某个角色在某个镜头中的具体外观和提示词
CREATE TABLE script_shot_characters (
    id BIGSERIAL PRIMARY KEY,
    shot_id BIGINT NOT NULL REFERENCES script_shots(id) ON DELETE CASCADE,
    character_id BIGINT REFERENCES script_characters(id), -- 关联角色ID
    appearance_features TEXT,            -- 本镜头中的外观特征 (可能换装)
    
    -- 图片提示词拆解 (正面)
    prompt_front_pos TEXT,               -- 正面-正向提示词
    prompt_front_neg TEXT,               -- 正面-反向提示词
    img_front_url TEXT,                  -- 生成的正面图路径
    
    -- 图片提示词拆解 (背面)
    prompt_back_pos TEXT,
    prompt_back_neg TEXT,
    img_back_url TEXT,
    
    -- 图片提示词拆解 (侧面)
    prompt_side_pos TEXT,
    prompt_side_neg TEXT,
    img_side_url TEXT
);
COMMENT ON TABLE script_shot_characters IS '镜头中的角色详情与提示词';

-- 5. 镜头视觉场景表 (对应 visualScene)
-- 记录场景描述、运镜、景别以及场景本身的三视图(如果有)
CREATE TABLE script_shot_scenes (
    id BIGSERIAL PRIMARY KEY,
    shot_id BIGINT NOT NULL REFERENCES script_shots(id) ON DELETE CASCADE,
    scene_content TEXT,                  -- 场景内容描述
    shot_size VARCHAR(50),               -- 景别 (中景/全景)
    camera_movement VARCHAR(50),         -- 运镜 (固定镜头/跟随)
    
    -- 场景参考图/生成图路径 (对应 visualScene 中的 frontImage 等)
    scene_img_front TEXT,
    scene_img_back TEXT,
    scene_img_side TEXT
);
COMMENT ON TABLE script_shot_scenes IS '镜头场景视觉信息';

-- 6. 镜头音视频生成表 (对应 imageToVideoPrompts, video, audioPerformance)
-- 记录视频生成提示词、结果以及音频信息
CREATE TABLE script_shot_media (
    id BIGSERIAL PRIMARY KEY,
    shot_id BIGINT NOT NULL REFERENCES script_shots(id) ON DELETE CASCADE,
    
    -- 视频生成提示词
    video_prompt_pos TEXT,               -- 视频正向
    video_prompt_neg TEXT,               -- 视频反向
    video_url TEXT,                      -- 最终视频地址
    
    -- 音频/对白信息
    dialogue_content TEXT,               -- 对白内容
    voice_over TEXT,                     -- 旁白
    emotion VARCHAR(100),                -- 情感
    sound_effects VARCHAR(255)           -- 音效
);
COMMENT ON TABLE script_shot_media IS '镜头音视频生成参数及结果';
```

### 3. 接口路由设计 (RESTful API)

基础路由: `/api/v4/shot/`

| 方法 | 路由 | 描述 | 备注 |
| :--- | :--- | :--- | :--- |
| **POST** | `/import` | **导入剧本JSON** | 核心接口，解析复杂JSON并分发到上述6张表 |
| **GET** | `/list` | **获取镜头列表** | 支持分页、筛选（剧本库ID、文件ID） |
| **GET** | `/{id}` | **获取单个镜头详情** | 聚合所有关联表信息返回完整结构 |
| **PUT** | `/{id}` | **更新镜头信息** | 修改文本、提示词、状态等 |
| **DELETE** | `/{id}` | **删除镜头** | 级联删除关联数据 |
| **GET** | `/characters` | **获取剧本库角色列表** | 查看该剧本库有哪些已注册角色 |
| **POST** | `/{id}/generate/image` | **(动作) 生成图片** | 触发特定镜头特定视角的图片生成 |
| **POST** | `/{id}/generate/video` | **(动作) 生成视频** | 触发视频生成 |

### 5. 后端逻辑实现建议 (Go/Python伪代码逻辑)

在实现 `/import` 接口时，请遵循以下逻辑以确保数据正确关联：

1.  **开启事务** (Transaction)。
2.  **插入文件表**：将 `fileInfo` 插入 `script_files`，获取 `new_file_id`。
3.  **遍历 `scriptShotList`**：
    *   **插入 Shot**：插入 `script_shots`，关联 `new_file_id`，获取 `new_shot_id`。
    *   **处理角色 (重要)**：
        *   获取 `characterInfo.characterName`。
        *   查询 `script_characters` 表：`WHERE library_id = ? AND name = ?`。
        *   **如果有**：获取其 `character_id`。
        *   **如果没有**：插入新角色到 `script_characters`，并获取新ID。
        *   **插入 Shot_Character**：将 `characterInfo` 中的 `textToImagePrompts` (正/背/侧) 和 `appearanceFeatures` 插入 `script_shot_characters` 表，关联 `new_shot_id` 和 `character_id`。
    *   **插入 Scene**：将 `visualScene` 数据插入 `script_shot_scenes`，关联 `new_shot_id`。
    *   **插入 Media**：将 `imageToVideoPrompts`, `video`, `audioPerformance` 数据插入 `script_shot_media`，关联 `new_shot_id`。
4.  **提交事务**。

这样设计后，当你需要查询“某剧本库中‘林风眠’的所有镜头”时，只需通过 `script_characters` 获取ID，再联表查询即可，非常方便数据分析和可视化。
